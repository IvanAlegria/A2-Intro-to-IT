Project Idea

2D top-down shooter game using Unity

For this assessment task, groups of students were tasked to develop an IT project idea and expand on the plans our group will take in order to make the idea a reality for the upcoming assignment. The idea we plan develop is a ‘2d top-down shooter game using Unity’. The purpose of this project idea is to use programming languages such as JavaScript or C-sharp to develop a shooter game using the game development tool called Unity. The game will be a “shoot ‘em up” type in which the player combats many enemies by shooting at them whilst dodging their fire.

game would be placed in the ‘top-down’ genre (top-down shooters specifically) and positioned to target a specific audience who like fast-paced and bullet hell style games. For example, games such as “Nuclear Throne”, “Binding of Isaac”, and “Vampire Survivors” which are popular for its difficulty, rouge-like gameplay and is also known to be some of the best-selling games in the genre.

The motivation to commence this project was mainly my interest and passion for games whether it be a shooter genre, an RPG, if it was 2D or 3D. And by learning how to program, trying to develop a game would be interesting as I would be creating something that I like, along with learning how games are created through coding. Also, due to the lack of experience in game development creating a 2D would be ideal, as not only is it easier to make, but it would also be a great introduction to learn the fundamentals.

to why I specifically want to build this game, through recently conducted research I have found that developing a 2d game would be the best project as a beginner to game development. As it is much less complicated than 3d games which would require additional processes such as 3d modelling. Additionally, by implementing a pixel game style would make it easier to create game assets such as the player model, and also provide a minimalistic and simple game that is appealing in the top-down shooter genre.


Description

As mentioned, the project is a 2D top-down shooter game similar to games such as “Enter the Gungeon” and “Nuclear Throne”. The objective of the game is to collect as many ‘boxes’ whilst fighting hordes of enemies and dodging returning fire. By collecting these ‘boxes’ the player will be accumulating points and will also receive a new random weapon from a pool. As points continue to increase so will the rate at which enemies spawn and how difficult they will be to kill. The main selling point of this game is that it is fast-paced and its repeatability. As players continue to play in order to achieve a new personal best in terms of points by collecting boxes whilst also killing and dodging incoming enemy fire provides the audience the experience of a bullet-style game. and as the game continues to become harder the more points accumulated, it makes breaking one’s personal record even more rewarding and satisfying. The final developed game should include the features:


  • Working player model
  • o Player model orientation follows the mouse cursor
  • o Player model moves when directional keys are pressed
  • o Model animations; running, idle, damaged, etc.
  • Gameplay
  • o Can fire gun by clicking the left mouse button
  • o Gun shoots towards mouse cursor
  • o ‘Boxes’ are randomly generated throughout the map
  • o When player picks up a box, the point counter increases
  • o Boxes disappears when picked up
  • o Players current weapon is replaced with a random weapon when picking up a box
  • o Various weapons with different features
  • Enemies
  • o Enemy number of enemies that spawn increases when the point counter goes up
  • o Different enemy types and health points
  • o Different enemy attack/shooting patterns
  • Map
  • o Enemies have a set spawn point in the map
  • o The map has walls in which players can take cover from
  • o Walls have collision (players can’t walk or shoot through)
  • Other
  • o Gameplay sounds (BGM, gun sounds, hit sounds, etc.)

Tools and Technology

To develop the game is mainly Unity, which is a free 2D/3D engine that provides a system for designing games and or applications. This software will be used to program using either C-sharp or JavaScript and run the game itself.

Our team plans to learn and use the programming language C-sharp to develop our game. Using unity as our main program to create most of the required items for the game such as the main menu, player movement, enemy pathing, etc. We will also be using Unity to play test our application to check for any error or bugs during the development process by creating an early test build in which will be shared to the whole team to test. The reason why our team has decided to utilise Unity instead of other game development application is because that its beginner friendly with many tutorials and also provides many features to make 2d game creation easier which can be helpful during the development of our project idea.

Unity has an asset store in which users can download free or paid assets that can be imported to the program, Unity does not enable users to model 2D or 3D assets. Therefore software such as Blender or Photoshop will be required to design the player, enemy, gun, effects, and map designs. Audio software such as "Ardour" or "LMMS which are both free and can be used to make the project’s in-game music.

Lastly, once the game has gone through multiple test-builds, fixed, and is ready to be exported, we will be exporting the game through Unity’s build function in which will create a folder with all the games contents. Anyone who has this folder will be able to then play the game by executing the exe. File located in the folder.

Implementation plans

Character Classes

By having a variation of character classes in our game would greatly improve the games replayability and encourage users to keep playing. It would also allow players to learn what kind of playstyle suits them by having different class abilities from a ‘assassin’ like class where the player does massive damage but have very little hp (health points), to a ‘tank’ like character where they are hard to kill but output significantly less damage to enemies. Doing so will make players want to play the game more to learn about each classes abilities and learn how to utilise it in specific scenarios in game. To implement this idea, we would need to research and learn how to code several classes which have their own specialised skill data. In addition to creating the 2d models and moving animation for each class.


Character Upgrades

A feature that can be implemented is a character upgrade mechanic where when a player collects a specific amount of ‘boxes’ they are given a variety of upgrades in which they can choose one upgrade that will be inherited by their in game character. The requirements of boxes needed is then increased to choose another upgrade. Having this mechanic will allow players to experience playthroughs that are different from the other as the upgrade are random, users are unlikely to have the exact same upgrade combination to previous playthroughs. Doing so will further increase replayability and ensure that players do not get bored of the same levels repeatedly.

However, due to the lack of knowledge and experience in coding it will be difficult to implement in the game prototype. The input of time will also be a massive factor on whether this feature can truly be implemented.


Unique 2D Jumping Mechanic

For our 2D top-down shooter game a possible mechanic that would separate it from other similar titles in the market could come from the implementation of a vertical jump mechanic. Whilst many top-down shooters are accustomed to a dodge or roll mechanic, we are yet to come across a vertical jump. In game this would see the playable character able to jump/vault over walls in the defined play area. These walls in-game enemies would not be able to pass through, instead having to gradually walk around said obstacle giving the character time. This jump mechanic could be implemented to a specific ‘jumping class’ character or alternatively be a foundational game wide mechanic. 

the way we would implement this into the game, the most likely solution would to be by selecting a ‘jump’ input key, which when pressed commences a sequence that has both a visual and mechanical element: 

1. Visual Element: given this is a 2D top-down shooter, when the jump button is pressed the sprite graphic, a small circle for this explanation, would be programmed to change into numerous incrementally larger circles (giving the visual illusion the player is jumping up), then shifting back down to the original circle size (giving the impression the character is falling back down). This animation sequence would only occur in around a 2 second window and look as if the head of the character is coming closer to the screen (viewer), mimicking the illusion of a jump from a bird’s eye view. 

2. Mechanical/Functionality Element: When the jump button is pressed we will program the character sprite to be able to pass through/over certain hitboxes (such as walls and other obstacles). Alternatively, we could set a height map across the whole playing area that has different Y-axis values for certain objects (e.g. Floor = Y-axis 1, Wall = y-axis 3). We would then program the player to detect the height map by binding the jump key to allow the player to access these different height values (e.g. only can access y-axis 1 - but when jump key is actioned player can access y-axis 3 momentarily. This, paired with the visual animation will give the player the impression that their character has just jumped over/onto the given obstacle.

Ageing Mechanic

This game mechanic introduces players into a looping system that has the character upon death enter an ageing mechanic. This was taken out of a game which similarly implemented element being Sifu. The brawler style fix camera game has players battling against a variety of enemy groups with the character starting in their teens. The more times players approach death, the game takes an interesting spin, allowing the character to age, coming back stronger with less health. Thus, introducing a balancing mechanic that utilizes replay ability as the same level can be approached at different “age” save points, developing additional tactics to beat the game. However, as the main story is not yet developed, it is hard to vision if this will be implemented for the team’s game.

Loot Box System

A possible gameplay mechanic would involve the boxes that the player would pick up within run. I would contain one from a pool of many weapons that have unique gameplay elements such as different weapon types (eg. melee, pistols, rifles, single shots), burn damage, even different projectile types, or other aspects that the group may come up with.
